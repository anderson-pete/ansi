# Contributing

I'm happy if this project is useful to you, but I want to be upfront: I'm selective about outside contributions. This is a personal project with a specific vision, and I maintain it on my own time.

## Before opening a PR

Open an issue first and describe what you want to change and why. I'll let you know whether it's something I'm likely to accept before you invest time writing code. PRs that arrive without prior discussion may not receive due consideration.

## What I'm likely to accept

- Bug fixes with a clear, minimal reproduction case
- Documentation improvements
- Tests
- Performance improvements that don't change behavior or add complexity
- Fixes for genuine correctness issues (wrong output, spec violations)

## What I'm unlikely to accept

- New features I didn't ask for
- Refactors or style changes
- Dependency additions
- Changes that expand scope beyond the project's purpose

## Code style

Read [`CODE_STYLES.md`](CODE_STYLES.md) before writing any code. PRs that don't follow it will be asked to revise.
